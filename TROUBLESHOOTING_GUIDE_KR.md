# 🛠️ 문제 해결 및 연결 정상화 가이드

현재 발생하고 있는 **데이터베이스 접속 불능** 및 **애플리케이션 설정 오류**를 해결하기 위해 순서대로 진행해야 할 작업 리포트입니다.

---

## 1단계: 네트워크 연결성 확인 (가장 중요)

제공해주신 DB 서버(`125.129.62.102:1521`)에 대한 네트워크 접근 권한이 없는 상태입니다.

1.  **Telnet 테스트**: 터미널(PowerShell 또는 CMD)에서 아래 명령어를 실행합니다.
    ```powershell
    telnet 125.129.62.102 1521
    ```
    - **결과 확인**: "연결하지 못했습니다"라고 나오면 네트워크가 차단된 것입니다.
2.  **IP 화이트리스트 요청**: DB 관리자에게 연락하여 **본인의 공인 IP**가 해당 DB 서버의 **1521 포트**에 접근 가능하도록 방화벽 해제를 요청해야 합니다.

---

## 2단계: 백엔드(Backend) 서버 정상화

포트 충돌로 인해 백엔드가 제대로 작동하지 않을 수 있습니다.

1.  **기존 프로세스 정리**: 터미널에서 아래 명령어로 포트 8000을 점유 중인 프로세스를 강제 종료합니다.
    ```powershell
    taskkill /f /im python.exe
    ```
2.  **백엔드 재시작**: `backend` 디렉토리에서 아래 명령어로 서버를 실행합니다 (8000 포트 고정).
    ```powershell
    cd backend
    uvicorn main:app --port 8000 --host 0.0.0.0
    ```

---

## 3단계: 프론트엔드(Frontend) 설정 고정

프론트엔드가 백엔드를 올바른 경로로 찾아가도록 설정해야 합니다.

1.  **API_BASE 확인**: `frontend-app/src/App.tsx` 파일의 11번 라인을 확인합니다.
    - **올바른 설정**: `const API_BASE = "/api";`
    - **주의**: `http://localhost:8080/api` 처럼 직접 포트를 적으면 프록시 오류가 발생할 수 있으므로 절대 경로를 사용하지 마세요.
2.  **Proxy 설정 확인**: `frontend-app/vite.config.ts` 파일에서 타겟이 `http://127.0.0.1:8000`으로 되어 있는지 확인합니다.

## 4단계: 브라우저 캐시 및 재접속

설정을 변경한 후에는 브라우저에 남아있는 이전 설정값을 비워야 합니다.

1.  **강제 새로고침**: 브라우저에서 `http://localhost:5173` 접속 후 **Ctrl + F5**를 눌러 캐시를 새로고침합니다.
2.  **연결 테스트**: 1단계에서 네트워크 차단이 해제된 것을 확인한 후, 화면에서 접속 정보를 입력하고 `Test Connection` 버튼을 누릅니다.

## 5단계: Docker 및 백그라운드 모니터링 (PowerShell)

문제를 진단하기 위해 컨테이너와 백그라운드 프로세스의 상태를 확인하는 방법입니다.

### 1. Docker 로그 및 상태 확인
컨테이너가 내부에서 어떤 에러를 겪고 있는지 확인합니다.
- **실행 중인 컨테이너 목록 확인**:
  ```powershell
  docker ps
  ```
  > [!NOTE] 
  > **목록이 비어있나요?**
  > 이 프로그램의 마이그레이션 러너(`ora2pg-runner`)는 실시간으로 작업을 처리할 때만 잠깐 생성되었다가, 작업이 끝나면 자동으로 삭제(`--rm` 옵션)됩니다. 따라서 마이그레이션 도중이 아니라면 목록에 나타나지 않는 것이 정상입니다.
- **컨테이너 실시간 로그 확인** (예: `ora2pg-runner` 또는 `postgres` 컨테이너):
  ```powershell
  docker logs -f [컨테이너_ID_또는_이름]
  ```
- **컨테이너 내부 셸 접속**:
  ```powershell
  docker exec -it [컨테이너_ID_또는_이름] /bin/bash
  ```

### 2. 백그라운드 프로세스 및 포트 확인
- **특정 포트(예: 8000)를 쓰는 프로세스 ID(PID) 찾기**:
  ```powershell
  netstat -ano | findstr :8000
  ```
- **실행 중인 Python 프로세스 모두 확인**:
  ```powershell
  Get-Process -Name python* | Select-Object Id, ProcessName, StartTime
  ```
- **파일 로그 실시간 모니터링** (Linux의 `tail -f`와 동일):
  ```powershell
  Get-Content -Wait [로그파일명.txt]
  ```

## 6단계: Docker로 띄우는 PostgreSQL 서버의 원리

PostgreSQL을 Docker로 실행할 때의 핵심 메커니즘 3가지는 다음과 같습니다.

1.  **격리 (Isolation)**: 호스트 OS(Windows)와 독립된 리눅스 환경에서 DB가 실행됩니다. 호스트에 PG를 직접 설치하지 않아도 되므로 환경이 깔끔하게 유지됩니다.
2.  **포트 매핑 (Port Mapping)**: `-p 5432:5432` 옵션은 **"호스트의 5432 포트로 들어오는 신호를 컨테이너 내부의 5432 포트로 전달하라"**는 뜻입니다. 이 설정이 없으면 외부에서 컨테이너 내부 DB에 접속할 수 없습니다.
3.  **볼륨 마운트 (Volume Mounting)**: `-v` 옵션을 통해 호스트의 폴더와 컨테이너 내부 폴더를 연결합니다. 컨테이너를 삭제하더라도 실제 데이터 파일은 호스트에 남아있어 데이터가 사라지지 않습니다.

**PostgreSQL 컨테이너 실행 예시 (참고용):**
```powershell
docker run --name dev-postgres -e POSTGRES_PASSWORD=pass123 -p 5432:5432 -d postgres
```

## 7단계: DDL Preview(변환 미리보기) 실패 시 확인 로그

객체 선택 화면에서 "Preview"를 눌렀을 때 변환이 안 된다면, 다음 세 곳의 로그를 확인해야 합니다.

### 1. 백엔드(Python) 터미널 로그 (가장 중요)
미리보기 버튼을 누르면 백엔드 창에 `[DDL Conversion]`으로 시작하는 로그가 출력됩니다.
- **확인 내용**:
  - `Docker exit code: 1`: 도커 내부에서 ora2pg가 실패했다는 뜻입니다.
  - `FAILED: DPY-3010: ...`: 파이썬이 DB로부터 원본 DDL을 가져오지 못했다는 뜻입니다 (네트워크 차단 가능성 99%).

### 2. 브라우저 개발자 도구 (F12)
- **Console 탭**: `Failed to fetch` 또는 `500 Internal Server Error`가 떠 있는지 확인하세요.
- **Network 탭**: `/api/convert/ddl` 요청의 응답(Response) 탭을 보면 백엔드에서 보낸 상세 에러 메시지가 들어있습니다.

### 3. 왜 미리보기가 실패하나요?
미리보기는 단순한 텍스트 변환이 아니라 **실시간으로 DB에 접속**하는 과정입니다.
1.  **Backend**가 Oracle에 접속해서 원본 DDL을 읽어옵니다.
2.  **Docker 컨테이너**가 다시 Oracle에 접속해서 `ora2pg` 엔진을 돌립니다.
두 과정 중 하나만 네트워크 문제로 실패해도 미리보기가 나오지 않습니다. 현재처럼 **1521 포트가 차단된 상태**라면 미리보기 기능도 작동하지 않는 것이 정상입니다.
